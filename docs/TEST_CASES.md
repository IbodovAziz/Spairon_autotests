# Критические тест-кейсы Spairon

Активный production smoke-набор проверяет только стабильные критические
функции:

- авторизацию;
- клиентскую валидацию формы входа;
- регистрацию без создания аккаунта;
- выход из аккаунта;
- статус и доступность интеграции HeadHunter.

Вакансии, отклики, анализ кандидатов, биллинг, комментарии и метки временно не
входят в CI-набор, пока соответствующая функциональность дорабатывается.

## Принципы набора

- Каждый кейс реализован отдельным тестом.
- Тесты не зависят от результата другого теста.
- Каждый тест получает новый browser context.
- Авторизованный кейс самостоятельно выполняет вход.
- Общий setup-проект и сохранённый `storageState` не используются.
- Запрещённые изменяющие запросы к production блокируются fixture.
- Регистрация проверяется только до отправки валидной формы.

## Авторизация

Источник: `tests/critical/public/authentication.spec.ts` и
`tests/critical/authenticated/session.spec.ts`.

### AUTH-001 — Страница входа доступна

- Открыть `/login`.
- Проверить HTTP `200`, title, заголовок, поля email/пароля и кнопку входа.
- При падении отчёт сообщает, что именно отсутствует или какой HTTP-статус
  получен.

### AUTH-002 — Email обязателен

- Оставить email пустым, заполнить пароль и нажать «Войти».
- Ожидается сообщение «Укажите email».
- `POST /auth/login` не должен отправляться.

### AUTH-003 — Некорректный email отклоняется

- Ввести `invalid-email` и тестовый пароль.
- Ожидается сообщение «Некорректный email».
- Пользователь остаётся на `/login`.

### AUTH-004 — Пароль обязателен

- Ввести корректный по формату email, оставить пароль пустым.
- Ожидается сообщение «Укажите пароль».
- `POST /auth/login` не должен отправляться.

### AUTH-005 — Успешный вход

- Ввести `SPAIRON_TEST_EMAIL` и `SPAIRON_TEST_PASSWORD`.
- Проверить `POST /auth/login → 200`.
- Проверить переход на `/vacancies` и наличие `accessToken`.

## Регистрация

Источник: `tests/critical/public/registration.spec.ts`.

### REG-001 — Страница регистрации доступна

- Открыть `/register`.
- Проверить HTTP `200`, title, заголовок, кнопку и поля:
  имя, организация, email, телефон, пароль, подтверждение пароля.

### REG-002 — Имя обязательно

- Отправить пустую форму.
- Ожидается сообщение «Укажите имя».

### REG-003 — Email обязателен

- Заполнить имя и оставить email пустым.
- Ожидается сообщение «Укажите email».

### REG-004 — Некорректный email отклоняется

- Заполнить имя и ввести `invalid-email`.
- Ожидается сообщение «Некорректный email».

### REG-005 — Телефон обязателен

- Заполнить имя и email, оставить телефон пустым.
- Ожидается сообщение «Укажите телефон».

### REG-006 — Пароль обязателен

- Заполнить имя, email и телефон, оставить пароль пустым.
- Ожидается сообщение «Укажите пароль».

### REG-007 — Пароли должны совпадать

- Заполнить обязательные поля и ввести разные пароли.
- Ожидается сообщение «Пароли не совпадают».

Валидная регистрация не выполняется: production-тест не должен создавать
пользователей. Для полного happy path регистрации нужен отдельный disposable
аккаунт или тестовое окружение.

## Выход

Источник: `tests/critical/authenticated/session.spec.ts`.

### AUTH-006 — Успешный выход

- Самостоятельно выполнить вход.
- Открыть «Настройки» → «Аккаунт».
- Нажать «Выйти».
- Проверить `POST /auth/logout → 200/204`.
- Проверить переход на `/login` и удаление `accessToken`.
- Проверить, что `/vacancies` снова перенаправляет на `/login`.

## HeadHunter

Источник: `tests/critical/authenticated/hh-integration.spec.ts`.

### HH-001 — API статуса интеграции доступен

- Самостоятельно выполнить вход.
- Выполнить `GET /hh-integration/status`.
- Ожидается HTTP `200`, JSON и boolean-поля `connected`, `isExpired`.
- Для подключённой интеграции проверяются `connectedAt` и `expiresAt`.

### HH-002 — OAuth-ссылка формируется

- Самостоятельно выполнить вход.
- Выполнить `GET /hh-integration/authorize-url` с уникальным `state`.
- Ожидается HTTP `200` и HTTPS-ссылка на домен `hh.ru`.
- Возвращённый `state` должен совпадать с отправленным.

### HH-003 — Раздел интеграции доступен

- Самостоятельно выполнить вход.
- Получить фактический статус HH через API.
- Открыть настройки и вкладку «Интеграция».
- UI должен соответствовать API:
  - подключено — статус и кнопка отключения;
  - не подключено — кнопка подключения.
- Кнопки изменения интеграции не нажимаются.

## Понятные ошибки

Названия тестов и шагов написаны на русском и содержат ID кейса. Для каждого
assertion задано прикладное сообщение.

GitHub Summary и Telegram показывают название и непосредственную причину:

```text
[AUTH-005] Пользователь может войти в аккаунт
Причина: Авторизация не выполнена: POST /auth/login вернул HTTP 401.
Ответ: Неверный email или пароль | Expected: 200 | Received: 401
```

Для упавшего теста также сохраняются screenshot, video, trace и HTML-отчёт.

## Команды

Весь активный критический набор:

```powershell
npm run test:critical
```

Только публичные проверки:

```powershell
npm run test:public
```

Только вход, выход и HeadHunter:

```powershell
npm run test:authenticated
```

Показать список тестов без запуска:

```powershell
npx playwright test --list
```
