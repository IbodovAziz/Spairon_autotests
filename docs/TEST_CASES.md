запуска.

## Обозначения

- **P0** — критичный smoke-тест. Падение означает недоступность основной
  функции или нарушение защиты данных.
- **P1** — важная проверка пользовательского интерфейса или навигации.
- **Passed** — ожидаемый результат получен.
- **Skipped** — тест не запускался из-за отсутствующего предусловия.
- **Failed** — ожидаемый результат не получен.

## Публичные страницы авторизации

Источник: `tests/public/auth-pages.spec.ts`.

### PUB-AUTH-001 — Отображение и обязательность полей входа

- **Приоритет:** P0.
- **Назначение:** проверить доступность формы входа и клиентскую валидацию без
  отправки авторизационного запроса.
- **Шаги:**
  1. Открыть `/login`.
  2. Проверить title, заголовок, email и password.
  3. Нажать «Войти» с пустыми полями.
  4. Проверить сообщение «Укажите email».
  5. Ввести синтаксически корректный email, оставить пароль пустым.
  6. Повторно нажать «Войти».
- **Ожидаемый результат:** отображается сообщение «Укажите пароль»; запрос на
  вход с паролем не отправляется.
- **Фактический результат:** **Passed**, 2,2 с.

### PUB-AUTH-002 — Ссылки со страницы входа

- **Приоритет:** P1.
- **Назначение:** проверить доступность восстановления, регистрации и
  юридических документов.
- **Шаги:** открыть `/login` и проверить адреса четырёх ссылок.
- **Ожидаемый результат:**
  - «Забыли пароль?» ведёт на `/forgot-password`;
  - «Зарегистрироваться» ведёт на `/register`;
  - политика ведёт на `https://spairon.ru/privacy-policy`;
  - оферта ведёт на `https://spairon.ru/terms-of-service`.
- **Фактический результат:** **Passed**, 1,6 с.

### PUB-AUTH-003 — Валидация регистрации без создания аккаунта

- **Приоритет:** P0.
- **Назначение:** проверить форму регистрации, не создавая пользователя в
  production.
- **Шаги:**
  1. Открыть `/register`.
  2. Проверить title и заголовок.
  3. Отправить пустую форму.
  4. Проверить сообщение «Укажите имя».
  5. Ввести имя и некорректный email.
  6. Повторно нажать «Зарегистрироваться».
- **Ожидаемый результат:** отображается «Некорректный email»; регистрационный
  запрос не отправляется.
- **Фактический результат:** **Passed**, 1,7 с.

### PUB-AUTH-004 — Валидация восстановления пароля

## Авторизованные read-only тесты

Источники:

- `tests/auth.setup.ts`;
- `tests/authenticated/core-readonly.spec.ts`.

Предусловия всего набора:

- `.env` находится рядом с `package.json`;
- заданы `SPAIRON_TEST_EMAIL` и `SPAIRON_TEST_PASSWORD`;
- используется выделенная production-учётная запись;
- для полного покрытия у аккаунта есть хотя бы одна существующая вакансия.

### AUTH-SETUP-001 — Подготовка авторизованной сессии

- **Приоритет:** P0.
- **Назначение:** подтвердить работоспособность входа и сохранить сессию для
  остальных тестов.
- **Шаги:**
  1. Открыть `/login`.
  2. Ввести email и пароль из `.env`.
  3. Нажать «Войти».
  4. Проверить переход на `/vacancies`.
  5. Проверить заголовок вакансий.
  6. Сохранить browser storage в `playwright/.auth/user.json`.
- **Ожидаемый результат:** вход успешен, открыта страница вакансий, сессия
  сохранена.
- **Фактический результат:** **Passed**, 3,1 с.

### AUTH-CORE-001 — Загрузка страницы вакансий

- **Приоритет:** P0.
- **Назначение:** проверить основной экран авторизованного пользователя.
- **Шаги:** открыть `/vacancies`, проверить URL, заголовок и отсутствие ошибки
  загрузки.
- **Ожидаемый результат:** страница вакансий отображается без сообщения «Не
  удалось загрузить вакансии».
- **Фактический результат:** **Passed**, 2,1 с.

### AUTH-CORE-002 — Чтение критичных API аккаунта

- **Приоритет:** P0.
- **Назначение:** проверить доступность основных backend-функций действующей
  сессии.
- **Шаги:**
  1. Получить access token текущей сессии.
  2. Выполнить авторизованные GET-запросы:
     `/auth/me`, `/vacancies`, `/hh-integration/status`, `/billing/balance`.
- **Ожидаемый результат:** каждый endpoint возвращает `200` и JSON.
- **Фактический результат:** **Passed**, 3,7 с.

### AUTH-CORE-003 — Открытие существующей вакансии

- **Приоритет:** P0.
- **Назначение:** проверить доступность критического экрана вакансии и анализа
  без запуска нового анализа.
- **Шаги:**
  1. Использовать `SPAIRON_TEST_VACANCY_ID` или первую существующую вакансию.
  2. Открыть `/vacancies/{id}/analysis`.
  3. Проверить URL.
  4. Проверить отсутствие ошибок загрузки вакансии и анализа.
- **Ожидаемый результат:** существующая вакансия открывается; сообщения «Не
  удалось загрузить вакансию» и «Не удалось загрузить анализ» отсутствуют.
- **Фактический результат:** **Passed**, 2,7 с.
- **Примечание:** если у аккаунта нет вакансий, тест получает статус
  **Skipped**, а не **Failed**.

## Защита production

Источник: `tests/fixtures/production-test.ts`.

Для UI smoke-тестов разрешены только `GET`, `HEAD`, `OPTIONS` и технический
`POST /auth/refresh`. Любой другой `POST`, `PUT`, `PATCH` или `DELETE` к
production API блокируется браузером и приводит к падению теста.

Автотесты не должны:

- создавать пользователей или демо-вакансии;
- запускать или перезапускать анализ;
- создавать платежи;
- добавлять, менять или удалять заметки и метки;
- отключать или изменять HeadHunter-интеграцию.

## Команды запуска

Публичные тесты:

```powershell
npm test
```

Авторизованные тесты:

```powershell
npm run test:authenticated
```

Все проекты одним запуском:

```powershell
npm run test:all
```

Открытие последнего HTML-отчёта:

```powershell
npm run report
```
