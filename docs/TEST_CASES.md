# Тест-кейсы Spairon

Документ описывает smoke-автотесты production-приложения
[app.spairon.ru](https://app.spairon.ru), назначение каждого теста,
проверяемый сценарий и результат контрольного запуска.

| Всего | 31 | 31 passed |

Общее время единого запуска: **56,5 секунды**.

Окружение контрольного запуска:

- frontend: `https://app.spairon.ru`;
- API: `https://server.spairon.ru`;
- Node.js: `24.14.0`;
- npm: `11.9.0`;
- Playwright: `1.60.0`;
- браузер: Chromium;
- worker: `1`, без параллельной нагрузки на production.

Статус в этом документе является снимком конкретного запуска. Актуальный
результат всегда следует смотреть в консоли или HTML-отчёте после нового
запуска.

## Обозначения

- **P0** — критичный smoke-тест. Падение означает недоступность основной
  функции или нарушение защиты данных.
- **P1** — важная проверка пользовательского интерфейса или навигации.
- **Passed** — ожидаемый результат получен.
- **Skipped** — тест не запускался из-за отсутствующего предусловия.
- **Failed** — ожидаемый результат не получен.

## Защита production

Источник: `tests/fixtures/production-test.ts`.

Для UI smoke-тестов разрешены только `GET`, `HEAD`, `OPTIONS` и технический
`POST /auth/refresh`. Любой другой `POST`, `PUT`, `PATCH` или `DELETE` к
production API блокируется браузером и приводит к падению теста.

Авторизованные GET-запросы к одинаковым endpoint кэшируются в пределах одного
worker. При `429 Too Many Requests` helper учитывает `Retry-After` и выполняет
не более двух повторов. Это снижает нагрузку на production.

Автотесты не должны:

- создавать пользователей или демо-вакансии;
- запускать или перезапускать анализ;
- создавать платежи;
- добавлять, менять или удалять заметки и метки;
- отключать или изменять HeadHunter-интеграцию.

## Команды запуска

```

Авторизованные тесты:

```

Все проекты одним запуском:

```powershell
npm run test:all
```

Открытие последнего HTML-отчёта:
